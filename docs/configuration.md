# Configuration System

The Open Data Coop uses a layered configuration system for managing settings and secrets.

## Quick Start

1. **Create your local config**:
   ```bash
   cp config.example.yml config.local.yml
   ```

2. **Create your environment file**:
   ```bash
   cp .env.example .env
   ```

3. **Edit your settings**:
   - Edit `config.local.yml` for pipeline settings and paths
   - Edit `.env` for secrets (API tokens, passwords, etc.)

4. **Test your configuration**:
   ```bash
   pixi run python scripts/test_config.py
   ```

## Configuration Files

### config.example.yml

Contains all available configuration options with sensible defaults. This file is:
- Checked into git
- Used as the base configuration
- Safe to share (no secrets)

**Do NOT edit this file directly.** Instead, create `config.local.yml` to override values.

### config.local.yml

Your personal configuration that overrides settings from `config.example.yml`. This file is:
- **NOT** checked into git (in `.gitignore`)
- Optional (falls back to example config if missing)
- Merged with example config (deep merge)

Example `config.local.yml`:
```yaml
# Override only what you need
paths:
  duckdb: "my_warehouse.duckdb"
  paimon_warehouse: "~/my-data/warehouse"

green_button:
  enabled: true
  xml_directory: "/Users/me/greenbutton_files"
  default_home_id: "my-home-123"

home_assistant:
  enabled: true
  parquet_file: "/Users/me/ha_export.parquet"
```

### .env

Environment variables for secrets. This file is:
- **NOT** checked into git (in `.gitignore`)
- Used for API tokens, passwords, credentials
- Loaded automatically by config system

Example `.env`:
```bash
# InfluxDB credentials
INFLUXDB_URL=http://localhost:8086
INFLUXDB_TOKEN=my-secret-token
INFLUXDB_ORG=my-org
INFLUXDB_BUCKET=home_assistant

# Green Button files
GREEN_BUTTON_DIR=/path/to/xml/files

# Cloud credentials
AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
```

### .env.example

Template for `.env` file. Shows all available environment variables. Checked into git.

## Configuration Features

### Environment Variable Expansion

Use `${VAR_NAME}` syntax in config files to reference environment variables:

```yaml
paths:
  duckdb: "${DUCKDB_PATH}"  # Reads from environment

home_assistant:
  parquet_file: "${HA_PARQUET_FILE}"
```

If the environment variable is not set, the config system will raise an error (except when the entire value is just the env var reference, in which case it returns `None`).

### Home Directory Expansion

Use `~` for home directory in paths:

```yaml
paths:
  paimon_warehouse: "~/energy-data/warehouse"
  # Expands to: /Users/yourname/energy-data/warehouse
```

### Deep Merge

When you create `config.local.yml`, only the values you specify are overridden. All other values come from `config.example.yml`.

Example:
```yaml
# config.example.yml
paths:
  duckdb: "energy_warehouse.duckdb"
  logs: "logs"
  data: "data"

# config.local.yml
paths:
  duckdb: "my_warehouse.duckdb"

# Merged result
paths:
  duckdb: "my_warehouse.duckdb"  # from local
  logs: "logs"                   # from example
  data: "data"                   # from example
```

## Using Configuration in Code

### Load configuration

```python
from pipelines.config import load_config

# Load config
config = load_config()

# Access values
duckdb_path = config['paths']['duckdb']
warehouse_dir = config['paths']['paimon_warehouse']
```

### Use global config instance

```python
from pipelines.config import get_config

# Get cached config (loads once)
config = get_config()
```

### Type-safe path access

```python
from pipelines.config import load_config
from pathlib import Path

config = load_config()

# Get path with expansion
warehouse_path: Path = config.get_path('paths.paimon_warehouse')
```

## Configuration Sections

### paths

File and directory paths:
- `duckdb`: DuckDB database file
- `mock_data`: Directory for test/mock data
- `paimon_warehouse`: Export directory for Paimon warehouse
- `logs`: Log file directory
- `data`: Data file directory

### home_assistant

Home Assistant power monitoring pipeline:
- `enabled`: Whether to enable pipeline
- `parquet_file`: Path to parquet export file
- `influxdb`: InfluxDB connection settings (optional)

### green_button

Green Button utility data pipeline:
- `enabled`: Whether to enable pipeline
- `xml_directory`: Directory with Green Button XML files
- `default_home_id`: Default home identifier
- `file_pattern`: Glob pattern for XML files

### paimon

Paimon warehouse export configuration:
- `exports`: List of tables to export with partition columns

### dlt

Data Load Tool (dlt) settings:
- `pipeline_names`: Names for dlt pipelines
- `destination`: Target destination (duckdb, postgres, etc.)
- `datasets`: Schema/dataset names

### quality

Data quality settings:
- `max_watts`: Maximum acceptable power reading
- `filter_negative`: Filter negative values
- `log_warnings`: Log data quality warnings

### logging

Logging configuration:
- `level`: Log level (DEBUG, INFO, WARNING, ERROR)
- `format`: Log message format
- `log_to_file`: Write logs to file
- `log_file`: Log file path

## Security Best Practices

1. **Never commit secrets to git**
   - Always use `.env` for secrets
   - Never put tokens in config files

2. **Use environment variables for sensitive data**
   ```yaml
   # Good
   influxdb:
     token: "${INFLUXDB_TOKEN}"

   # Bad
   influxdb:
     token: "my-secret-token-12345"
   ```

3. **Share example files, not actual configs**
   - Share: `config.example.yml`, `.env.example`
   - Don't share: `config.local.yml`, `.env`

4. **Use separate configs per environment**
   ```bash
   # Development
   cp config.example.yml config.local.yml

   # Production
   cp config.example.yml config.production.yml
   ```

## Troubleshooting

### "Environment variable 'X' referenced in config but not set"

**Solution**: Add the variable to your `.env` file:
```bash
echo "X=value" >> .env
```

### "Config file not found"

**Solution**: Create `config.local.yml` or ensure `config.example.yml` exists in project root.

### "Required config field 'X' is missing"

**Solution**: Check that `config.example.yml` has the required field, or update your `config.local.yml`.

### Changes to config not taking effect

**Solution**: If using `get_config()`, restart your Python process. The config is cached globally.

For testing, use `reset_config()`:
```python
from pipelines.config import reset_config, get_config

reset_config()  # Clear cache
config = get_config()  # Reload
```
