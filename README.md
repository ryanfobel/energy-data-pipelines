**Privacy-first, open-source infrastructure for residential energy data**

This repository contains projects and proposals for building cooperative energy data infrastructure that puts homeowners in control of their own consumption data while enabling powerful analysis for heat pumps, solar, and rate optimization.

## Vision

Ontario's Green Button mandate gives every electricity customer the legal right to share their hourly interval data with authorized third parties, but almost no accessible, privacy-respecting infrastructure exists to make this useful for homeowners. This project aims to change that through open-source tools and cooperative structures.

### Core Principles

1. **Privacy-first**: Data encrypted with user's own keys; never stored in cloud databases
2. **Local-first**: High-resolution data stays under user control (Home Assistant, local servers)
3. **Open source**: All tools are freely available and auditable
4. **Cooperative model**: Infrastructure incubated through community organizations (e.g., TREC, CED Co-op)
5. **Reproducible**: Documented deployments that scale beyond pilots

## Proposals

### Green Button Data Platform

[`docs/proposals/green-button-pilot.md`](docs/proposals/green-button-pilot.md)

Pilot proposal for a privacy-first Green Button data platform targeting CED Co-op members (Kitchener-Waterloo region).

**Note:** While the Green Button API infrastructure is being built, [utility-bill-scraper](https://github.com/ryanfobel/utility-bill-scraper) provides an immediate solution for Kitchener residents to download monthly bills and track emissions.

**What it enables for homeowners:**
- **Solar sizing** from real hourly consumption (not annual kWh estimates)
- **Rate optimization** — model ULO vs TOU savings for EVs and heat pumps
- **Heat pump performance** — measure actual COP over time using degree days
- **Retrofit readiness** — auditors assess exact usage patterns, not guesses from bills

**Pilot ask:** 30-50 CED Co-op members, 3-month trial, zero ongoing effort from Co-op

### Home Energy Digital Twin

[`docs/proposals/home-energy-digital-twin.md`](docs/proposals/home-energy-digital-twin.md)

Full technical proposal for a residential heat pump monitoring and optimization platform.

**Architecture:**
- Home Assistant for local data collection (power, temperature, thermostat)
- Periodic sync to cloud platform (AWS/GCP/self-hosted)
- Time-series database (InfluxDB, TimescaleDB, or DuckDB)
- AI energy coaching interface (Claude API)

**Tiered data approach:**
- **T0 (Required):** Utility bills + weather data + electricity pricing
- **T1 (Common):** Green Button hourly data
- **T2 (Valuable):** HOT2000 energy audit model + real-time power monitoring
- **T3 (Enhanced):** Temperature sensors + smart thermostat integration
- **T4 (Advanced):** Supply/return temperatures + thermal storage sensors

**MVP:** 5-home pilot during 2025-26 heating season to validate infrastructure and establish heat pump performance baselines.

## Technical Architecture

### Data Format: Apache Paimon

The platform uses **Apache Paimon** as the recommended data format for home energy data warehouses:

**Why Paimon:**
- **Fully file-based** — No catalog server required; works with object storage, local filesystems, or network drives
- **Branching support** — Version control for data; experiment with different models without affecting production
- **File-based sync** — Mirror between home data warehouse and co-op backup using rsync/rclone
- **Incremental processing** — Efficient updates from continuous IoT sensor streams
- **Open table format** — No vendor lock-in; integrates with DuckDB, Spark, Flink, Trino

**Architecture:**
```
Home Assistant → Paimon (local) → rsync/rclone → Paimon (co-op backup)
                    ↓
                DuckDB/Ibis queries
                    ↓
                Personal dashboards
```

**Governance:** Data stays on user devices by default. Co-op backup is opt-in with encryption. Aggregation for research uses differential privacy (implementation TBD).

## Related Ecosystem Components

This project builds on and integrates with several open-source energy data tools:

- **greenbutton-python** — XML parsing for Green Button data (multiple implementations evaluated)
- **[homeassistant-helpers](https://github.com/ryanfobel/homeassistant-helpers)** — IoT data hub for local monitoring (InfluxDB, MQTT, Parquet exports)
- **[electricitymap-contrib](https://github.com/electricitymap/electricitymap-contrib)** — Real-time carbon intensity of electricity grids

## Use Cases

### For Homeowners

1. **Solar sizing** — Use your actual hourly load shape across seasons, not utility estimates
2. **Heat pump selection** — Compare performance against your real heating load
3. **Rate plan optimization** — Model exactly what switching to TOU or ULO would save with your EV/heat pump
4. **Energy audit validation** — Verify actual consumption against HOT2000 model predictions
5. **Retrofit ROI** — Quantify savings from insulation, windows, or HVAC upgrades

### For Energy Cooperatives

1. **Member benefits** — Offer data tools without building/maintaining infrastructure
2. **Community aggregation** — Privacy-preserving analysis across member cohorts
3. **Policy advocacy** — Evidence-based arguments for better rates, programs, and regulations
4. **Education** — Help members understand their energy use patterns

### For Researchers

1. **Heat pump field data** — Real-world COP measurements across diverse housing stock
2. **Grid interaction** — Impact of residential electrification on demand curves
3. **Model validation** — Test HOT2000 predictions against measured consumption
4. **Program evaluation** — Assess retrofit programs with normalized savings calculations

## Getting Started

### Quick Start (5 minutes)

Run your own privacy-first energy data pipeline in 5 minutes:

```bash
# 1. Fork and clone
git clone https://github.com/YOUR_USERNAME/open-data-coop.git
cd open-data-coop

# 2. Install dependencies with pixi
curl -fsSL https://pixi.sh/install.sh | bash  # Install pixi if needed
pixi install

# 3. Generate mock data and run the pipeline
pixi run python scripts/generate_mock_ha_data.py
pixi run python scripts/test_full_pipeline.py

# 4. Query your data warehouse
pixi run duckdb energy_warehouse.duckdb
```

**That's it!** You now have:
- Mock energy data loaded into DuckDB
- dbt transformations applied
- A queryable data warehouse with electricity consumption facts
